## 1. Homebrew 安装

### 1.1 安装 Homebrew

```bash
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
```

### 1.2 配置环境变量

安装完成后，需要将 Homebrew 添加到 PATH：

```bash
echo >> /Users/ugreen/.zprofile
echo 'eval "$(/opt/homebrew/bin/brew shellenv zsh)"' >> /Users/ugreen/.zprofile
eval "$(/opt/homebrew/bin/brew shellenv zsh)"
```

### 1.3 验证安装

```bash
brew --version
# 输出：Homebrew 5.1.0

which brew
# 输出：/opt/homebrew/bin/brew

brew --prefix
# 输出：/opt/homebrew
```

### 1.4 安装目录结构

Homebrew 安装在 `/opt/homebrew` 目录下：

- **`/opt/homebrew/bin`** - 可执行文件
- **`/opt/homebrew/Cellar`** - 已安装的软件包
- **`/opt/homebrew/Caskroom`** - macOS 应用程序
- **`/opt/homebrew/etc`** - 配置文件
- **`/opt/homebrew/lib`** - 库文件
- **`/opt/homebrew/share`** - 共享文件

## 3. Git 多账号管理

当需要在同一台电脑上使用多个 Git 账号（如工作账号和个人账号）时，可以通过目录配置和 SSH 密钥管理来实现自动切换。

### 3.1 配置方案说明

本方案采用 **目录配置 + SSH 密钥** 的组合方式：
- 根据项目所在目录自动使用对应的 Git 账号信息
- 使用不同的 SSH 密钥区分不同的账号

**示例配置：**
- 工作项目目录：`~/UGproject/`
- 个人项目目录：`~/Iproject/`

### 3.2 步骤一：配置 Git 全局配置文件

创建 `~/.gitconfig` 文件：

```bash
[user]
    name = zhupanfeng
    email = zhupanfeng@ugreen.com

# 工作项目目录自动使用工作账号
[includeIf "gitdir:~/UGproject/"]
    path = ~/.gitconfig-work

# 个人项目目录自动使用个人账号
[includeIf "gitdir:~/Iproject/"]
    path = ~/.gitconfig-personal
```

### 3.3 步骤二：创建独立的账号配置文件

**工作账号配置** (`~/.gitconfig-work`)：

```bash
[user]
    name = zhupanfeng
    email = zhupanfeng@ugreen.com
```

**个人账号配置** (`~/.gitconfig-personal`)：

```bash
[user]
    name = ilili
    email = jupowfer@gmail.com
```

### 3.4 步骤三：生成 SSH 密钥

为不同账号生成独立的 SSH 密钥：

```bash
# 创建 .ssh 目录（如果不存在）
mkdir -p ~/.ssh && chmod 700 ~/.ssh

# 生成工作账号的 SSH 密钥
ssh-keygen -t ed25519 -C "zhupanfeng@ugreen.com" -f ~/.ssh/id_ed25519_work -N ""

# 生成个人账号的 SSH 密钥
ssh-keygen -t ed25519 -C "jupowfer@gmail.com" -f ~/.ssh/id_ed25519_personal -N ""
```

### 3.5 步骤四：配置 SSH config 文件

创建 `~/.ssh/config` 文件：

```bash
# 工作 GitHub 账号
Host github-work
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_ed25519_work

# 个人 GitHub 账号
Host github-personal
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_ed25519_personal

# 工作 GitLab 账号
Host gitlab-work
    HostName gitlab.com
    User git
    IdentityFile ~/.ssh/id_ed25519_work

# 个人 GitLab 账号
Host gitlab-personal
    HostName gitlab.com
    User git
    IdentityFile ~/.ssh/id_ed25519_personal

# 工作 Gitee 账号
Host gitee-work
    HostName gitee.com
    User git
    IdentityFile ~/.ssh/id_ed25519_work

# 个人 Gitee 账号
Host gitee-personal
    HostName gitee.com
    User git
    IdentityFile ~/.ssh/id_ed25519_personal
```

设置正确的权限：

```bash
chmod 600 ~/.ssh/config
```

### 3.6 步骤五：添加 SSH 公钥到 Git 平台

#### 查看公钥

```bash
# 查看工作账号公钥
cat ~/.ssh/id_ed25519_work.pub

# 查看个人账号公钥
cat ~/.ssh/id_ed25519_personal.pub
```

#### 添加到 GitHub

1. 登录对应的 GitHub 账号
2. 进入 **Settings** → **SSH and GPG keys** → **New SSH key**
3. 粘贴对应的公钥内容
4. 点击 **Add SSH key**

#### 添加到 GitLab/Gitee

步骤类似，在对应平台的 SSH 密钥设置页面添加公钥。

### 3.7 步骤六：验证配置

#### 验证目录配置

```bash
# 在工作目录下验证
cd ~/UGproject/UGTest/UGDemo
git config user.name
# 输出：zhupanfeng
git config user.email
# 输出：zhupanfeng@ugreen.com

# 在个人目录下验证
cd ~/Iproject/test
git config user.name
# 输出：ilili
git config user.email
# 输出：jupowfer@gmail.com
```

#### 验证 SSH 连接

添加公钥后，测试 SSH 连接：

```bash
# 测试工作账号连接
ssh -T git@github-work
# 成功输出：Hi username! You've successfully authenticated...

# 测试个人账号连接
ssh -T git@github-personal
# 成功输出：Hi username! You've successfully authenticated...
```

### 3.8 使用方法

#### 克隆仓库

**工作项目（使用工作账号）：**

```bash
cd ~/UGproject
git clone git@github-work:company/repo.git
```

**个人项目（使用个人账号）：**

```bash
cd ~/Iproject
git clone git@github-personal:username/repo.git
```

#### 修改已有仓库的 remote

如果已有仓库需要切换账号：

```bash
# 查看当前 remote
git remote -v

# 修改为工作账号
git remote set-url origin git@github-work:company/repo.git

# 修改为个人账号
git remote set-url origin git@github-personal:username/repo.git
```

### 3.9 常见问题

#### Q1: 如何查看当前使用的配置？

```bash
# 查看当前用户信息
git config user.name
git config user.email

# 查看配置来源
git config --show-origin user.email
```

#### Q2: 已经提交了错误的用户信息怎么办？

```bash
# 修改最后一次提交的作者信息
git commit --amend --author="正确的名字 <correct@email.com>"

# 修改多个提交（谨慎使用）
git rebase -i HEAD~3  # 修改最近3个提交
# 在每个需要修改的提交前将 pick 改为 edit
# 执行 git commit --amend --author="..." 后继续 git rebase --continue
```

#### Q3: SSH 连接失败怎么办？

```bash
# 检查 SSH 密钥是否已添加到 ssh-agent
ssh-add -l

# 如果没有，手动添加
ssh-add ~/.ssh/id_ed25519_work
ssh-add ~/.ssh/id_ed25519_personal

# 测试连接并查看详细信息
ssh -vT git@github-work
```

### 3.10 配置文件位置总结

- **Git 全局配置**: `~/.gitconfig`
- **工作账号配置**: `~/.gitconfig-work`
- **个人账号配置**: `~/.gitconfig-personal`
- **SSH 配置**: `~/.ssh/config`
- **SSH 密钥**:
  - 工作私钥：`~/.ssh/id_ed25519_work`
  - 工作公钥：`~/.ssh/id_ed25519_work.pub`
  - 个人私钥：`~/.ssh/id_ed25519_personal`
  - 个人公钥：`~/.ssh/id_ed25519_personal.pub`

## 4. JDK 安装与版本管理

### 4.1 查看可用的 JDK 版本

```bash
brew search openjdk
```

输出示例：
```
==> Formulae
openjdk          openjdk@17       openjdk@8
openjdk@11       openjdk@21
```

### 4.2 安装 JDK

```bash
# 安装 JDK 17
brew install openjdk@17

# 安装 JDK 21
brew install openjdk@21
```

### 4.3 安装 jenv（JDK 版本管理工具）

```bash
brew install jenv
```

### 4.4 配置 jenv

将 jenv 添加到 shell 配置文件：

```bash
echo 'export PATH="$HOME/.jenv/bin:$PATH"' >> ~/.zshrc
echo 'eval "$(jenv init -)"' >> ~/.zshrc
source ~/.zshrc
```

### 4.5 将 JDK 添加到 jenv

```bash
# 添加 JDK 17
jenv add /opt/homebrew/opt/openjdk@17/libexec/openjdk.jdk/Contents/Home

# 添加 JDK 21
jenv add /opt/homebrew/opt/openjdk@21/libexec/openjdk.jdk/Contents/Home
```

### 4.6 设置全局 JDK 版本

```bash
# 设置 JDK 21 为全局默认版本
jenv global 21
```

### 4.7 清理重复版本

jenv 添加 JDK 时会创建多个别名，可以删除重复的：

```bash
# 查看所有版本
jenv versions

# 删除重复的别名，只保留主版本号
jenv remove 17.0
jenv remove 17.0.18
jenv remove openjdk64-17.0.18
jenv remove 21.0
jenv remove 21.0.10
jenv remove openjdk64-21.0.10
```

清理后的版本列表：
```
  system
  17
* 21 (set by /Users/ugreen/.jenv/version)
```

### 4.8 验证 JDK 配置

```bash
java -version
# 输出：
# openjdk version "21.0.10" 2026-01-20
# OpenJDK Runtime Environment Homebrew (build 21.0.10)
# OpenJDK 64-Bit Server VM Homebrew (build 21.0.10, mixed mode, sharing)
```

## 5. Maven 安装与版本管理

### 5.1 为什么选择 SDKMAN？

Homebrew 只提供最新版本的 Maven（如 3.9.14），无法安装特定的旧版本（如 3.9.2）。SDKMAN 是专门用于管理 JVM 生态工具版本的工具，支持安装任意历史版本。

### 5.2 安装 Bash 5+（SDKMAN 依赖）

macOS 默认使用 Bash 3.2，SDKMAN 需要 Bash 4+：

```bash
brew install bash
```

### 5.3 安装 SDKMAN

使用新安装的 Bash 运行安装脚本：

```bash
/opt/homebrew/bin/bash -c "$(curl -s https://get.sdkman.io)"
```

### 5.4 配置 SDKMAN

安装完成后，SDKMAN 会自动添加配置到 `~/.zshrc`。需要加载 SDKMAN 环境：

```bash
source ~/.sdkman/bin/sdkman-init.sh
```

或者重启终端。

### 5.5 安装 Maven 3.9.2

```bash
# 查看所有可用的 Maven 版本
sdk list maven

# 安装 Maven 3.9.2
sdk install maven 3.9.2

# 设置为默认版本
sdk default maven 3.9.2
```

### 5.6 验证 Maven 安装

```bash
mvn --version
# 输出：
# Apache Maven 3.9.2 (c9616018c7a021c1c39be70fb2843d6f5f9b8a1c)
# Maven home: /Users/ugreen/.sdkman/candidates/maven/current
# Java version: 21.0.10, vendor: Homebrew
```

## 6. Python 安装与版本管理

### 6.1 为什么选择 pyenv？

macOS 系统自带 Python，但版本较旧且不建议修改系统 Python。pyenv 是 Python 版本管理的首选工具，可以：
- 安装任意 Python 版本（包括 CPython、PyPy、Anaconda 等）
- 轻松切换全局、项目级、会话级的 Python 版本
- 避免与系统 Python 冲突

### 6.2 安装 pyenv

```bash
brew install pyenv
```

### 6.3 配置 pyenv

将 pyenv 添加到 shell 配置文件：

```bash
echo 'export PYENV_ROOT="$HOME/.pyenv"' >> ~/.zshrc
echo '[[ -d $PYENV_ROOT/bin ]] && export PATH="$PYENV_ROOT/bin:$PATH"' >> ~/.zshrc
echo 'eval "$(pyenv init -)"' >> ~/.zshrc
source ~/.zshrc
```

### 6.4 查看可用的 Python 版本

```bash
pyenv install --list
```

### 6.5 安装 Python

```bash
# 安装 Python 3.11
pyenv install 3.11.9

# 安装 Python 3.12
pyenv install 3.12.4

# 安装最新稳定版
pyenv install 3.13.0
```

### 6.6 设置 Python 版本

```bash
# 设置全局默认版本
pyenv global 3.12.4

# 在当前目录设置项目级版本（生成 .python-version 文件）
pyenv local 3.11.9

# 当前终端临时使用某个版本
pyenv shell 3.13.0
```

### 6.7 验证 Python 配置

```bash
python --version
# 输出：Python 3.12.4

which python
# 输出：/Users/ugreen/.pyenv/shims/python

pyenv versions
# 输出：
#   system
#   3.11.9
# * 3.12.4 (set by /Users/ugreen/.pyenv/version)
#   3.13.0
```

### 6.8 配置 pip 镜像源

#### 临时使用镜像源

```bash
pip install <package> -i https://pypi.tuna.tsinghua.edu.cn/simple
```

#### 永久配置镜像源


- 中国科技大学：`https://pypi.mirrors.ustc.edu.cn/simple`

### 6.9 使用虚拟环境（venv）

Python 3.3+ 内置 venv 模块，无需额外安装 virtualenv。

#### 创建虚拟环境

```bash
# 在当前目录创建 venv 目录
python -m venv venv

# 指定 Python 版本创建（确保该版本已通过 pyenv 安装）
python3.11 -m venv venv
```

#### 激活虚拟环境

```bash
# 激活
source venv/bin/activate

# 激活后提示符会显示 (venv)
# (venv) user@hostname:~$
```

#### 退出虚拟环境

```bash
deactivate
```

#### 删除虚拟环境

```bash
rm -rf venv
```

### 6.12 使用 uv（高性能包管理工具）

[uv](https://github.com/astral-sh/uv) 是由 Astral 团队用 Rust 编写的 Python 包管理工具，可替代 pip + venv + pip-tools，速度提升 10-100 倍。

#### 安装 uv

```bash
brew install uv
```

#### 配置国内镜像源

创建或编辑 `~/.config/uv/uv.toml`（推荐）或在项目根目录创建 `uv.toml`：

```toml
[pip]
index-url = "https://pypi.tuna.tsinghua.edu.cn/simple"
```

或使用环境变量：

```bash
export UV_INDEX_URL=https://pypi.tuna.tsinghua.edu.cn/simple
```

#### uv 常用命令

```bash
# 创建虚拟环境（默认在项目根目录创建 .venv）
uv venv

# 指定 Python 版本创建
uv venv --python 3.12

# 激活虚拟环境
source .venv/bin/activate

# 退出虚拟环境
deactivate

# 安装包（自动检测虚拟环境）
uv pip install <package>

# 安装指定版本
uv pip install <package>==<version>

# 从 requirements.txt 安装
uv pip install -r requirements.txt

# 生成 requirements.txt
uv pip freeze > requirements.txt

# 升级包
uv pip install --upgrade <package>

# 卸载包
uv pip uninstall <package>

# 列出已安装包
uv pip list

# 显示包信息
uv pip show <package>
```

#### 使用 uv 进行依赖锁定

```bash
# 从 pyproject.toml 生成锁定文件
uv pip compile pyproject.toml -o requirements.txt

# 从锁定文件安装（精确复现环境）
uv pip sync requirements.txt
```

#### 项目结构示例（使用 uv）

```

├── src/                # 源代码

├── requirements.txt    # 锁定依赖
```

## 7. Rust 安装与版本管理

### 7.1 为什么选择 rustup？

Rust 官方推荐使用 rustup 作为工具链管理器，可以：
- 安装和管理多个 Rust 版本（stable、beta、nightly）
- 管理编译目标（cross-compilation targets）
- 管理 Rust 组件（rustfmt、clippy、rust-analyzer 等）
- 自动更新 Rust 工具链

> **注意：** 不建议通过 Homebrew 安装 Rust，因为 Homebrew 只提供最新 stable 版本，无法管理多版本和组件。

### 7.2 安装 rustup

```bash
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
```

安装过程中会提示选择安装选项，直接按 `1` 选择默认安装即可。

安装完成后，rustup 会自动在 `~/.zshrc`（或 `~/.zprofile`）中添加环境变量配置：

```bash
. "$HOME/.cargo/env"
```

加载环境变量使其立即生效：

```bash
source ~/.cargo/env
```

### 7.3 验证安装

```bash
rustc --version
# 输出：rustc 1.xx.x (xxxxxxx 2026-xx-xx)

cargo --version
# 输出：cargo 1.xx.x (xxxxxxx 2026-xx-xx)

rustup --version
# 输出：rustup 1.xx.x (xxxxxxx 2026-xx-xx)
```

### 7.4 安装目录结构

rustup 和 cargo 的默认安装路径：

- **`~/.rustup/`** - rustup 管理的工具链和组件
  - `toolchains/` - 已安装的 Rust 工具链
  - `settings.toml` - rustup 配置
- **`~/.cargo/`** - Cargo 包管理器相关
  - `bin/` - 可执行文件（rustc、cargo、rustfmt 等）
  - `registry/` - crates.io 的本地缓存
  - `config.toml` - Cargo 配置

### 7.5 工具链管理

#### 查看已安装的工具链

```bash
rustup toolchain list
# 输出：
# stable-aarch64-apple-darwin (default)
```

#### 安装其他版本

```bash
# 安装 nightly 版本
rustup toolchain install nightly

# 安装 beta 版本
rustup toolchain install beta

# 安装特定版本
rustup toolchain install 1.75.0
```

#### 切换默认工具链

```bash
# 设置全局默认
rustup default stable
```

### 7.7 配置 Cargo 国内镜像源

Rust 的包注册中心 crates.io 在国内访问较慢，可配置国内镜像加速。

#### 配置 sparse 协议镜像（推荐）

创建或编辑 `~/.cargo/config.toml`：

```toml
[source.crates-io]
replace-with = 'ustc'

[source.ustc]
registry = "sparse+https://mirrors.ustc.edu.cn/crates.io-index/"
```

常用国内镜像源：
- 中国科技大学：`sparse+https://mirrors.ustc.edu.cn/crates.io-index/`
- 清华大学：`sparse+https://mirrors.tuna.tsinghua.edu.cn/crates.io-index/`
- 字节跳动：`sparse+https://rsproxy.cn/crates.io-index/`

#### 配置 rsproxy 全套加速（镜像 + rustup）

字节跳动的 rsproxy 同时提供 crates.io 和 rustup 的镜像加速：

```toml
# ~/.cargo/config.toml
[source.crates-io]
replace-with = 'rsproxy-sparse'

[source.rsproxy-sparse]
registry = "sparse+https://rsproxy.cn/crates.io-index/"

[registries.rsproxy]
index = "https://rsproxy.cn/crates.io-index"

[net]
git-fetch-with-cli = true
```

同时配置 rustup 镜像（添加到 `~/.zshrc`）：

```bash
export RUSTUP_DIST_SERVER="https://rsproxy.cn"
export RUSTUP_UPDATE_ROOT="https://rsproxy.cn/rustup"
```

## 8. 版本管理方式对比

### 8.1 JDK 版本管理对比

#### **Homebrew + jenv 方式**（推荐）

**优点：**
- ✅ 版本切换简单（一条命令）
- ✅ 支持项目级版本管理（`.java-version` 文件）
- ✅ 自动处理 `JAVA_HOME` 和 `PATH`
- ✅ 多版本共存，易于维护
- ✅ Homebrew 统一管理更新

**缺点：**
- ❌ 需要安装额外工具（Homebrew + jenv）
- ❌ 有一定学习成本

**使用场景：**
```bash
# 全局使用 JDK 21
jenv global 21

# 在某个项目目录下使用 JDK 17
cd /path/to/project
jenv local 17

# 当前终端临时使用 JDK 17
jenv shell 17
```

#### **手动安装 + 环境变量方式**

**优点：**
- ✅ 完全控制，可安装任意版本
- ✅ 无额外依赖

**缺点：**
- ❌ 切换版本需要手动修改配置文件
- ❌ 容易出错
- ❌ 不支持项目级版本切换
- ❌ 维护成本高

**典型配置：**
```bash
# 在 ~/.zshrc 中
export JAVA_HOME=/Library/Java/JavaVirtualMachines/jdk-21.jdk/Contents/Home
export PATH=$JAVA_HOME/bin:$PATH

# 切换版本需要修改配置并重新加载
source ~/.zshrc
```

#### **对比表格**

| 特性 | Homebrew + jenv | 手动安装 + 环境变量 |
|------|----------------|-------------------|
| **安装难度** | 简单（一条命令） | 需要下载、解压、配置 |
| **版本切换** | 一条命令即时切换 | 需要修改配置文件并重载 |
| **项目级版本** | 支持（`.java-version`） | 不支持 |
| **多版本共存** | 原生支持 | 需要手动管理 |
| **更新维护** | `brew upgrade` | 手动下载新版本 |
| **灵活性** | 中等 | 高（可安装任意版本） |

- ✅ 完全控制，可安装任意版本

## 10. 推荐配置方案

### 10.1 适用场景

- **需要多个 Python 版本** → Homebrew + pyenv
- **追求极致的包管理速度** → uv（替代 pip + venv）
- **开发多个项目，需要频繁切换 JDK 版本** → Homebrew + jenv
- **需要特定版本的 Maven** → SDKMAN
- **Rust 开发** → rustup（官方推荐，唯一正确方式）
- **只用一个固定版本** → Homebrew 或手动安装
- **团队协作，需要统一环境** → pyenv + uv + jenv + SDKMAN + rustup（可通过配置文件共享）

### 10.2 当前环境总结

本文档配置的环境：

- **Git**: 2.53.0（通过 Homebrew）
- **Python**: 3.11、3.12 和 3.13（通过 Homebrew + pyenv 管理）
  - 当前激活：Python 3.12.4
- **Python 包管理器**: uv（通过 Homebrew 安装，替代 pip + venv）
- **JDK**: 17 和 21（通过 Homebrew + jenv 管理）
  - 当前激活：JDK 21
- **Maven**: 3.9.2（通过 SDKMAN 管理）
- **Rust**: stable（通过 rustup 管理）
  - 包含组件：rustfmt、clippy、rust-analyzer、rust-src

### 10.3 配置文件位置

- **Homebrew 配置**: `~/.zprofile`
- **pyenv 配置**: `~/.zshrc`
- **uv 配置**: `~/.config/uv/uv.toml` 或项目目录 `uv.toml`
- **jenv 配置**: `~/.zshrc`
- **SDKMAN 配置**: `~/.zshrc` 和 `~/.sdkman/`
- **Rust/Cargo 配置**: `~/.cargo/config.toml`
- **rustup 配置**: `~/.rustup/settings.toml`
- **Python 安装位置**: `~/.pyenv/versions/`
- **JDK 安装位置**: `/opt/homebrew/opt/openjdk@<version>/`
- **Maven 安装位置**: `~/.sdkman/candidates/maven/`
- **Rust 工具链位置**: `~/.rustup/toolchains/`
- **Cargo 可执行文件**: `~/.cargo/bin/`

```bash
# 或者重启终端
```

### 11.5 Maven 使用错误的 JDK 版本

**问题**: Maven 使用的 Java 版本与 `java -version` 不一致

**解决方案**:
```bash
# 确保 jenv 的 export 插件已启用
jenv enable-plugin export

# 验证 JAVA_HOME
echo $JAVA_HOME

# 重新加载环境
source ~/.zshrc
```

### 11.6 Rust/Cargo 命令找不到

**问题**: `rustc` 或 `cargo` 命令不可用

**解决方案**:
```bash
# 手动加载 Cargo 环境
. "$HOME/.cargo/env"

# 检查 PATH 中是否包含 .cargo/bin
echo $PATH | tr ':' '\n' | grep cargo

# 如果 .cargo/env 不存在，重新安装 rustup
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
```

### 11.7 Rust 编译依赖缺失

**问题**: 编译时报错缺少链接器或系统库

**解决方案**:
```bash
# 确保已安装 Xcode Command Line Tools
xcode-select --install

# 如果已安装但仍报错，重新安装
sudo rm -rf /Library/Developer/CommandLineTools
xcode-select --install
```
